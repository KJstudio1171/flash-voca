# i18n / 로컬라이제이션 시스템 설계

- **작성일**: 2026-04-13
- **상태**: 설계 확정, 구현 계획 대기
- **관련 문서**: `docs/design-system.md`, `docs/superpowers/specs/2026-04-10-error-management-design.md`

## 1. 목적과 범위

이 설계는 flash-voca 앱에 다국어 지원 기반을 마련한다. 두 축의 국제화를 동시에 다룬다.

1. **앱 UI 다국어화** — 메뉴·버튼·라벨·에러 메시지 등 화면 텍스트
2. **학습 콘텐츠 다국어화** — 스토어에 진열되는 공식 번들/덱의 메타데이터(제목·설명)

### 1.1 지원 로케일

1차 지원: **`ko`, `en`, `ja`, `zh`** (4개). 폴백 체인은 `요청 로케일 → en → ko`.

### 1.2 범위에 포함되는 것

- 앱 UI 정적 문자열 번역
- `AppError.userMessage` 번역 (에러 메시지 국제화)
- 동적 포맷팅: 복수형, 날짜, 숫자, 상대 시간
- 공식 번들/덱 메타데이터 번역
- 기기 locale 자동 감지 + 설정 화면에서 수동 override

### 1.3 범위에서 제외되는 것

- 사용자 작성 콘텐츠(덱 제목, 카드 `term`/`meaning`)의 번역 — 항상 원본 표시
- RTL 언어 지원 — 1차 지원 로케일 모두 LTR
- 서버 측 번역 관리 도구 — 로케일 파일은 앱 번들에 포함, 공식 번들 번역은 seed/DB
- TTS/음성 합성 — 향후 과제

## 2. 3저장소 원칙

텍스트의 성격에 따라 저장 위치를 분리한다.

| 텍스트 종류 | 저장 위치 | 전환 시 동작 |
|---|---|---|
| UI 문자열 | `src/shared/i18n/locales/{ko,en,ja,zh}.json` | 즉시 재렌더 |
| 공식 번들/덱 메타 | `bundle_translations`, `deck_translations` 테이블 | Repository가 `locale` 파라미터로 조회, 없으면 원본 폴백 |
| 사용자 작성 | 기존 `decks.title`, `deck_cards.term/meaning` raw 컬럼 | 불변 — 번역 대상 아님 |

## 3. 아키텍처

### 3.1 계층도

```
Screen / Component
   │
   ├─ useT()        ── i18next ── locales/*.json
   ├─ useFormat()   ── Intl.*  ── 날짜/숫자/상대시간
   │
   └─ Service / Repository
                    │
                    └─ SQLite
                         ├─ bundle_translations / deck_translations (공식 메타)
                         └─ decks / deck_cards (사용자 콘텐츠 raw)
```

### 3.2 의존성

- **`i18next`** + **`react-i18next`** — 번역 엔진, React 훅
- **`expo-localization`** — 기기 locale 감지
- **`@react-native-async-storage/async-storage`** — 사용자 선택 로케일 영속화

### 3.3 composition root 연동

- `createAppServices.ts`에 `LocaleService` 추가
- `AppProviders.tsx`에 `I18nextProvider` 추가 → 트리 전역에서 `useT()` 동작
- `BootstrapService.prepareAppAsync`에 i18next 초기화 단계 추가 (DB 초기화와 병렬 가능)

## 4. 로케일 감지와 전환

### 4.1 최초 실행

```
앱 시작
  → LocaleService.resolveInitialLocale()
      1. AsyncStorage에서 userSelectedLocale 조회
      2. 있으면 그것을 사용
      3. 없으면 Localization.getLocales()[0].languageCode
      4. 지원 목록에 없으면 'en' 폴백
  → i18next.init({ lng: resolved, fallbackLng: ['en', 'ko'] })
  → AppBootstrapGate는 i18next 초기화 완료까지 대기
```

### 4.2 런타임 전환

```
설정에서 언어 선택
  → LocaleService.setLocale(next)
      1. AsyncStorage 저장
      2. i18next.changeLanguage(next) → useT 훅 전체 재렌더
      3. Intl 래퍼 캐시 무효화
      4. TanStack Query: locale 포함 쿼리 키가 변경되어 자동 refetch
```

### 4.3 설계 결정

- **i18next가 단일 진실 원본(SSOT)**. 현재 로케일 조회는 `i18n.language`를 사용.
- **`LocaleService`는 감지·영속화·변경의 진입점**만 담당.
- **TanStack Query 키 정책**: 공식 번들/덱 쿼리는 `['bundles', locale]` 형태로 locale 포함. 사용자 덱 쿼리는 locale 미포함.

## 6. 번역 키 컨벤션

- 1단계: 도메인/기능 — `common`, `errors`, `home`, `study`, `decks`, `store`, `profile`, `settings`
- 2단계: 구체 키 (camelCase)
- 복수형은 i18next 표준 `_one` / `_other` 서픽스
- 장문은 `screens.{name}.{section}.{purpose}`까지 허용

### 6.1 예시

```jsonc
{
  "common": { "save": "저장", "cancel": "취소", "retry": "다시 시도" },
  "home": {
    "title": "오늘의 학습",
    "dueCount_one": "{{count}}개 복습 예정",
    "dueCount_other": "{{count}}개 복습 예정"
  },
  "errors": {
    "network": "네트워크 연결을 확인해주세요",
    "database": "데이터를 불러올 수 없습니다"
  }
}
```

### 6.2 타입 안전성

- `ko.json`을 진실 원본으로 삼아 `TranslationKey` 유니온 타입 생성
- `useT()` 래퍼는 `TranslationKey`만 허용하여 오타·누락을 컴파일 타임 검출
- 다른 로케일 파일은 `ko.json`과 구조 일치(누락 허용, 여분 불허)

### 6.3 에러 메시지 연동

- `AppError`에 `messageKey: TranslationKey` + 선택적 `messageParams: Record<string, string | number>` 필드 추가
- `userMessage`는 `get userMessage()` 파생 속성으로 유지 (deprecated, 점진 제거)
- 렌더 계층은 `t(error.messageKey, error.messageParams)` 직접 호출 선호

## 7. DB 스키마 변경

### 7.1 `bundle_translations`

```sql
CREATE TABLE bundle_translations (
  bundle_id TEXT NOT NULL REFERENCES bundles(id) ON DELETE CASCADE,
  locale TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  PRIMARY KEY (bundle_id, locale)
);
CREATE INDEX idx_bundle_translations_locale ON bundle_translations(locale);
```

- `bundles.title` / `bundles.description` 은 **canonical 원본**으로 보존 — 최종 폴백
- 조회 패턴:
  ```sql
  SELECT b.*,
         COALESCE(bt.title, b.title) AS title,
         COALESCE(bt.description, b.description) AS description
  FROM bundles b
  LEFT JOIN bundle_translations bt
    ON bt.bundle_id = b.id AND bt.locale = ?
  ```

### 7.2 `deck_translations`

```sql
CREATE TABLE deck_translations (
  deck_id TEXT NOT NULL REFERENCES decks(id) ON DELETE CASCADE,
  locale TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  PRIMARY KEY (deck_id, locale)
);
```

`sourceType = 'official'` 덱만 번역 대상. 사용자 덱은 LEFT JOIN의 `COALESCE`가 원본을 반환하므로 동일 쿼리로 처리 가능.

### 7.3 Deck 언어 필드

```sql
ALTER TABLE decks ADD COLUMN source_language TEXT NOT NULL DEFAULT 'en';
ALTER TABLE decks ADD COLUMN target_language TEXT NOT NULL DEFAULT 'ko';
```

- 도메인 모델 `Deck`에 `sourceLanguage: LanguageCode`, `targetLanguage: LanguageCode` 추가
- `SaveDeckPayload`에 두 필드 필수 포함 → 덱 편집기에 언어 쌍 선택 UI 추가
- 카드 콘텐츠(`DeckCard.term/meaning`) 스키마는 변경 없음 — 언어 쌍은 부모 덱에서 상속

### 7.4 Repository 시그니처 변경

- `BundleRepository.listBundlesAsync(locale)`
- `BundleRepository.getBundleDetailAsync(id, locale)`
- `DeckRepository.listOfficialDecksAsync(locale)`
- 사용자 덱 메서드는 시그니처 변경 없음
- 상위 Service가 `LocaleService`에서 현재 로케일을 조회해 전달

### 7.5 마이그레이션

- 신규 마이그레이션 단계를 `src/core/database/initialize.ts`의 마이그레이션 파이프라인에 추가: 테이블 생성 → 컬럼 추가 순서 (구체적 버전 번호/파일명은 구현 시 기존 관례 따름)
- `seed.ts`: 공식 번들/덱 시드 시 4개 로케일 번역 행 동시 삽입
- 파괴적 변경 없음(ADD COLUMN / CREATE TABLE). 기존 사용자 없음(`LOCAL_USER_ID` 고정)이므로 무중단

## 8. 포맷팅 레이어

i18next는 텍스트 보간을 담당하고, 로케일 민감 포맷팅은 표준 `Intl` API를 얇게 래핑해 사용한다.

### 8.1 API

```ts
// format/dateFormat.ts
export function formatDate(
  date: Date | string,
  locale: LanguageCode,
  style?: 'short' | 'medium' | 'long'
): string;

// format/relativeTimeFormat.ts
export function formatRelativeTime(
  fromIso: string,
  locale: LanguageCode,
  now?: Date
): string;

// format/numberFormat.ts
export function formatNumber(value: number, locale: LanguageCode): string;
```

### 8.2 훅

```ts
export function useFormat() {
  const { i18n } = useTranslation();
  const locale = i18n.language as LanguageCode;
  return useMemo(() => ({
    date:     (d, style?) => formatDate(d, locale, style),
    relative: (iso)       => formatRelativeTime(iso, locale),
    number:   (n)         => formatNumber(n, locale),
  }), [locale]);
}
```

### 8.3 설계 결정

- `Intl.*` 인스턴스는 모듈 스코프 `Map<key, Intl.*>`로 캐시. `LocaleService.setLocale`이 캐시 무효화 이벤트 발행
- Expo 54 / Hermes는 `DateTimeFormat`, `NumberFormat`, `RelativeTimeFormat` 전부 기본 지원 — polyfill 불필요. 단, Android 실기기에서 `ja`/`zh` 출력 검증 필요
- 날짜는 DB에 UTC ISO로 저장 유지. 표시 시 `Intl`이 기기 시간대로 자동 변환
- i18next의 `{{date, datetime}}` 인라인 포맷은 **사용하지 않음** — 번역자 학습 비용과 타입 안전성 문제

## 9. 기존 코드 마이그레이션 전략

현재 13개 파일에 한국어가 하드코딩됨. 단계적으로 전환한다.

### 9.1 단계

1. **인프라 셋업** — 의존성 설치(`i18next`, `react-i18next`, `expo-localization`, `@react-native-async-storage/async-storage`), i18next 초기화, 로케일 파일 스켈레톤 4개, `useT`/`useFormat`, `TranslationKey` 생성 스크립트
2. **`common` + `errors` 네임스페이스 전환** — 공유 텍스트 먼저
3. **`AppError.messageKey` 도입** — `userMessage`는 파생 속성으로 공존
4. **화면별 전환** — `profile` → `settings` → `home` → `decks` → `store` → `study`
5. **공식 번들/덱 메타 이관** — seed + `bundle_translations` / `deck_translations` 동시 커밋, Repository 시그니처 변경
6. **덱 편집기 언어 쌍 UI** — `sourceLanguage` / `targetLanguage` 드롭다운
7. **설정 화면 언어 선택 UI** — 이전 단계까지는 기기 locale만 사용

각 단계는 독립 커밋/PR. 1~3 완료 시점까지는 사용자 가시 변화 없음.

### 9.2 번역본 채우기 정책

- 1차 릴리스 시 `ko.json` 100% 채움 (원본)
- `en.json` — 개발자 초벌 번역 후 검수
- `ja.json`, `zh.json` — 누락 키는 폴백 체인으로 표시. 빈 키 허용
- CI 검증: `ko.json`에만 없는 키(다른 파일의 여분 키) = 오타 가능성 → 실패

### 9.3 하드코딩 방지

- ESLint `i18next/no-literal-string` 규칙 추가, 전환 완료 파일에 opt-in 적용

### 9.4 롤백

- 신규 마이그레이션은 ADD COLUMN / CREATE TABLE만 수행 — 파괴적 변경 없음
- 롤백 필요 시 DROP / 컬럼 제거 마이그레이션을 별도 작성

## 10. 테스트 전략

### 10.1 계층별 범위

| 계층 | 테스트 | 로케일 처리 |
|---|---|---|
| `locales/*.json` | 키 구조·누락 검증 | 정적 분석 |
| `LocaleService` | 감지·영속화·전환 | 단위 테스트 |
| 포맷터 | 날짜/숫자/상대시간 | 로케일별 기대값 |
| Repository | `LEFT JOIN` + 폴백 | 인메모리 SQLite |
| Service | 로케일 전파 | 목 Repository |
| 화면 컴포넌트 | `t()` 결과 렌더 | `I18nextProvider` 래퍼 |
| `AppError` | `messageKey` 해석 | i18next 초기화 환경 |

### 10.2 로케일 파일 구조 검증 (자동)

- 모든 로케일이 `ko.json`의 키 집합의 부분집합인지 확인
- 누락 키는 경고(폴백 동작하므로 실패 아님)
- `ko.json`에 없는 여분 키는 실패

### 10.3 테스트 유틸리티

`__tests__/helpers/`에 추가:

- `renderWithI18n(ui, locale)` — 컴포넌트 테스트용 Provider
- `createBundleTranslation(overrides?)` — factories.ts 확장

### 10.4 Mock 정책

`CLAUDE.md` 규칙에 따라 `jest.mock()` 금지, 인터페이스 기반 수동 목. `LocaleService`는 `LocaleGateway` / `LocaleStorage` 협력자 인터페이스를 도입해 목 용이성 확보.

### 10.5 포맷터 스냅샷

각 로케일 × (단기 과거 / 장기 과거 / 미래) 조합. `Intl.RelativeTimeFormat` 출력은 런타임별 미세 변동 가능 → 느슨한 매처(정규식).

### 10.6 수동 검증 체크리스트

UI 변경 포함 PR은 Android 실기기에서:

- [ ] 설정에서 4개 로케일 전환, 모든 탭 순회
- [ ] 로케일 전환 후 앱 재시작 → 선택 유지 확인
- [ ] 기기 언어를 `ja`로 바꾸고 AsyncStorage 초기 상태로 첫 실행 → `ja` 적용
- [ ] 날짜/상대시간이 각 로케일 포맷으로 표시(예: `zh`의 "3天前")

## 11. 오픈 이슈

- 로케일 전환 시 `TanStack Query` 캐시 전체를 무효화할지, 로케일 포함 키만 자연 재조회에 맡길지 — 구현 단계에서 성능 측정 후 결정
- 설정 화면에서 로케일 변경 시 앱 재시작 유도 여부 — 핫 전환이 원칙이나 네이티브 모듈 관련 잔존 이슈가 발견되면 재시작 안내 UI 추가
